## Report structure

The special monthly matrix format is scoped to Department Reports only.

Header/document area:
- hospital / system identity
- report title
- selected department scope
- selected Thai month/year
- generated by / generated time
- landscape document metadata

Main matrix:
- `ลำดับ`
- `ชื่อ-สกุล`
- `ตำแหน่ง`
- `แผนก`
- day columns from `1` to the last day of the selected month

Footer:
- notes / legend
- signature line for `ผู้ตรวจสอบเวร`

## Day-column generation rules

- The matrix uses the selected department report month and year.
- Day columns are generated dynamically for:
  - 28 days
  - 29 days
  - 30 days
  - 31 days
- The selected Thai month/year shown in the heading is the same period used for matrix generation.

## Shift abbreviation mapping

- `ช` = เวรเช้า เวลา 08.30 - 16.30 น.
- `บ` = เวรบ่าย เวลา 16.30 - 00.30 น.
- `ด` = เวรดึก เวลา 00.30 - 08.30 น.
- `BD` = เวรบ่ายนอกเวลาราชการ

The matrix is built from real `time_logs` rows using shared helper logic in [report_helpers.php](C:/xampp/htdocs/staff-main/includes/report_helpers.php).

## Source fields used for mapping

Current schema does not expose a dedicated structured `shift_type` or preset-code column in `time_logs`, so the mapping uses the safest available source fields in this order:

1. approval state via `checked_at`
2. explicit BD marker in `note` or `approval_note`
3. exact normalized `time_in` + `time_out`

Only records with `checked_at IS NOT NULL` are eligible for the department monthly matrix.

## Time normalization rule

Before matching, the report normalizes time values to `HH:MM` 24-hour format.

Examples:

- `08:30:00` -> `08:30`
- `16:30:00` -> `16:30`
- `00:30:00` -> `00:30`

Seconds and extra formatting are ignored during matching.

## Exact mapping rules

Unless a stricter explicit BD marker overrides the result, the matrix uses these exact rules only:

- `08:30` -> `16:30` = `ช`
- `16:30` -> `00:30` = `บ`
- `00:30` -> `08:30` = `ด`

If a record does not match one of those exact normalized time ranges, it is not converted into a visible matrix code.

## BD detection rule

`BD` is mapped only by an explicit marker.

Accepted triggers in the current system:

- `note = BD`
- `approval_note = BD`
- text containing the explicit phrase `เวรบ่ายนอกเวลาราชการ` in `note` or `approval_note`

The report does not infer `BD` from generic afternoon time alone.

## Duplicate-day handling rule

If one person has multiple records on the same day, the monthly matrix resolves the cell with this strict priority:

1. any explicitly marked `BD` record -> `BD`
2. otherwise collect exact standard matches (`ช / บ / ด`)
3. if there is exactly one unique standard code -> use that code
4. if there are multiple conflicting standard codes on the same day -> blank cell

This avoids silently guessing when same-day records are ambiguous.

## Blank future-day rule

If the selected month is the current month and the day is later than today, the matrix cell is left blank.

- no `0`
- no placeholder symbol
- no fake shift code

## Blank-cell rule

The matrix cell is left blank when:

- no approved record exists for that person/day
- a record exists but does not match a safe mapping rule
- the date is in the future within the current incomplete month
- multiple same-day standard shift records conflict and cannot be resolved safely

## Report data scope rule

This monthly matrix reflects actual approved attendance/shift logs, not future planned roster assignments.

- approved-only source = `checked_at IS NOT NULL`
- future days in the current month remain blank

## Footer notes / signature structure

- `ด = เวรดึก เวลา 00.30 - 08.30 น.`

Signature section:

## A4 landscape print strategy

- Print layout uses A4 landscape.
- PDF generation uses landscape orientation as well.
- The layout removes dashboard-style body cards and gives priority to the matrix itself for formal paper use.
